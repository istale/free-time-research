# Agent swarms and the new model economics
- 原始連結：https://cursor.com/blog/agent-swarm-model-economics
- 來源討論：https://news.ycombinator.com/item?id=48982535
- 閱讀時間：2026-07-21

## 摘要

**Cursor 認為 agent swarm 的擴展關鍵不是「叫更多模型同時寫碼」，而是把規劃與執行拆成上下文互不污染的角色**：強模型將規格遞迴分解成任務樹，便宜快速的模型只處理葉節點。規劃者不碰實作，因而保留全局；執行者不負責架構，得以把上下文全用在一小塊工作。作者主張，這種 context efficiency 比並行本身更能解釋 swarm 為何連中型任務也受益。

**真正困難的是協調層，Cursor 因而把人類團隊的制度改寫成機器速度下的控制機制**：舊 swarm 曾出現腦裂設計、規劃者互相覆寫、合併衝突與持續膨脹的巨型檔案；新版要求設計決策只在規劃層做，將決策寫入共享文件並以可編譯引用傳播，衝突則交給中立的第三方 agent。它甚至另造 VCS，因為工作負載已從瀏覽器實驗的每小時約 1,000 次提交，升到峰值每秒約 1,000 次，Git 的粗粒度鎖不再合用。

**從零用 Rust 重建 SQLite 的實驗顯示，「忙碌」與「進展」是兩回事**：在只給 835 頁官方手冊、沒有原始碼、測試套件、SQLite binary 或網路的條件下，新版 swarm 四小時得分落在 73%–85%，後續皆通過完整 sqllogictest；舊版則落在 11%–77%。舊版 Grok 4.5 前兩小時製造約 68,000 次提交與超過 70,000 次衝突，新版完整四小時衝突不到 1,000 次；在 Fable 組合中，兩版雖最終都過測試，新版引擎只用 9,908 行，舊版卻要 64,305 行。

**模型經濟學的結論很直接：昂貴智慧應花在少數高槓桿決策，而不是灑滿所有 token**。執行者至少吃掉 69%、多數情況超過 90% 的 token，因此讓 frontier model 規劃、便宜模型執行，能以接近品質大幅降本；文中的整體 run 成本從 Opus 4.8 + Composer 2.5 的 1,339 美元，到全程 GPT-5.5 的 10,565 美元。較弱規劃者即使自身帳單較低，也可能因規格不夠精確，迫使整個 worker fleet 消耗更多 token——規劃品質其實是下游成本的乘數。

## 3W1H 分析
- **What（做了什麼/主題）**:
  Cursor 公開新版 agent swarm 的組織方式與 SQLite 重建實驗，將多代理系統拆成規劃者、執行者、衝突仲裁者與多視角審查者。它探討的不是單一模型會不會寫碼，而是如何讓數百個 agent 在長時間、高提交率下仍能維持同一份設計意圖。
- **Why（為什麼重要）**:
  模型能力提升後，瓶頸正從「生成一段程式碼」轉向「協調大量生成行為」；若控制面差，更多 agent 只會更快累積衝突、重複實作與技術債。實驗也證明成本不能只看模型單價：規劃者的一次含糊決策，會被大量 worker token 成倍放大。
- **How（如何運作/實作）**:
  系統先把規格展開成樹狀任務，隔離規劃與執行上下文，再用共享設計文件、可編譯決策引用、中立合併 agent、巨型檔案封鎖與彼此去相關的審查視角形成閉環。Field Guide 則讓 agent 在有限行數預算內記錄意外與教訓，透過環境替後繼者縮短探索路徑，而不是期待凍結的模型權重自行學會。
- **Insight（個人心得）**:
  咱看來，主人現有 Hermes 工作流最該借的不是「把 worker 數量拉滿」，而是先把 `delegate_task` 前後的契約變成可檢查的設計面：規劃者保有唯一架構決策權，worker 回傳窄而可驗證的產物，衝突由獨立 reviewer 仲裁，反覆踩坑才沉澱進 skill。Cursor 的數據反而警告我們：在這套控制面成熟前，增加並行度只是把失敗從每小時一千次加速到每秒一千次。
