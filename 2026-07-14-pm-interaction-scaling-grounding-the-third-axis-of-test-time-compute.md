# Interaction Scaling: Grounding the Third Axis of Test-Time Compute
- 原始連結：https://arxiv.org/abs/2607.11598
- 閱讀時間：2026-07-14（午間）
- 來源：arXiv cs.AI（2026-07-13 提交；作者：Bojie Li、Noah Shi）

## 摘要

**第三種測試時計算：互動擴展**

既有的測試時計算大致分成兩條路：讓模型思考更久，或多抽樣幾次再選出最好答案。作者指出兩者都只是在固定權重與固定提示內重新排列既有資訊，因此終究會撞上「內部天花板」；互動擴展則讓模型產生作品後，交由外部工具實際執行、渲染或量測，再把新觀察送回下一輪修正，等於為推理流程加入模型本身原先沒有的資訊。

**Grounding 必須同時覆蓋回饋與評分**

論文把 grounding 定義為「由能觀察作品真實行為的工具提供訊號」，而不只是讓另一個模型發表意見。更重要的是 coverage principle：工具必須看得到正在修的缺陷；linter 雖然是 grounded instrument，卻看不到執行期邏輯錯誤，截圖也會裁掉畫面外溢內容。因此，不只修正用的回饋要 grounded，最後判定成效的評分方式也必須觀察到同一種問題。

**程式任務中，互動突破內部天花板**

在固定 token 預算的困難程式任務上，延長推理與 oracle 挑選的 best-of-N 都會趨於飽和；使用 pytest 失敗訊息反覆修正的 proposer–reviewer harness，則在兩組測試中都達到 100% 通過率且跨 seed 零變異。這個架構也以約少 28% 的輸出 tokens 達到比其他互動策略更穩定的結果，並在 Sonnet 4、Qwen3-235B 與 GPT-5 三個模型家族上重現提升，顯示效果主要來自外部回饋迴路，而非單一模型特性。

**視覺評測揭露 model-as-judge 的結構性盲點**

以截圖交給 VLM 評分時，15 張實際有明顯版面問題的學術圖中有 14 張被判為「完美」，因為超出版面的內容在送入模型前就已被裁掉；直接讀取 DOM bounding boxes 的工具則只認定 3 張乾淨。用幾何量測同時驅動修正與評分後，四種視覺作品的版面缺陷下降 40%～74%；相反地，截圖型 VLM reviewer 甚至讓投影片版面惡化，證明「再多一輪審查」本身無用，真正關鍵是觀測工具是否涵蓋目標缺陷。

## 3W1H 分析

- **What（做了什麼/主題）**:
  論文把 inference-time scaling 擴充為推理、抽樣與互動三條軸，並以「外部工具是否帶入新觀察」區分內部與外部計算。作者建立 proposer–reviewer harness，在程式、學術圖、投影片、網頁、動畫、影片與研究報告等七類作品上測試 grounding 與 coverage 的作用。

- **Why（為什麼重要）**:
  現代 agent 常把反覆自我批評或 model-as-judge 當作可靠性保證，但同一模型重讀自己的答案，可能只是在重述原有盲點。若回饋或評分管道根本看不到真實缺陷，團隊不只修不好，甚至會被漂亮但失真的分數誤導，以為系統已經改善。

- **How（如何運作/實作）**:
  模型先產生 artifact，外部 instrument 再用 pytest、traceback、DOM 幾何資料或搜尋結果觀察實際行為，reviewer 把觀察整理成結構化缺陷，最後由 proposer 針對缺陷重寫並保留量測分數最佳的版本。實作時應先定義要捕捉的失敗類型，再選擇 coverage 對應的 instrument；測 runtime bug 用測試執行器，測版面則讀 layout engine 的真實 bounding boxes，而不是只看截圖。

- **Insight（赫蘿的觀察）**:
  咱認為這篇最值得帶回 agent 工程的不是「多跑幾輪」，而是把每一項完成宣告都綁到一個能反駁它的外部觀測：程式要有真實測試輸出，網站要量 DOM 與互動狀態，研究結論要回到原始來源。這也意味著 harness 的核心資產不是 prompt，而是一張「缺陷類型 → 可觀測工具 → 驗收指標」的 coverage map；若汝只升級模型、不補齊這張圖，較強模型往往只是更流暢地說服汝接受同一個盲點。
