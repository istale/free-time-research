# The Harness Effect: How Orchestration Design Sets the Token Economics of Enterprise Agentic AI
- 原始連結：https://arxiv.org/abs/2607.06906
- PDF：https://arxiv.org/pdf/2607.06906v1
- 閱讀時間：2026-07-10（午間）
- 來源：arXiv cs.AI 昨日新論文（v1 提交 2026-07-08，由 Writer, Inc. 33 位作者群掛名；最後作者 Waseem AlShikh 為公司 CTO）

## 摘要

**這是一篇由 Writer, Inc. 提出的 controlled-swap 自然實驗：用同一份 22 題 locked task set、同樣的 6 個 foundation model（Claude Sonnet 4.6 / Gemini 3.1 / Gemini Flash 3.5 / Qwen 3.6 / GLM 5.1 / Palmyra X6）、同樣的 judge panel、同樣的 pinned price table，只換 orchestration layer——「conventional production agent loop」vs.「Writer Agent Harness」——來量化「harness layer 對企業 agentic AI 總帳單的影響力」。**論文定義一個新詞：**token maxing**——用 tokens 換 capability（更長 reasoning trace、更多 turn、更寬 tool payload、更大 replayed context），tokens-per-task 的成長速度高於 task value 的成長速度，**這是個 Jevons 動態**（效率↑ → 單價↓ → 總量↑），per-token price 跌多少都救不了總支出的上升。

**核心 headline 數字（Section 6.1, blended across 6 models）：**換成 harness 後 cost-per-task −41%（$0.21 → $0.12）、median wall-clock −44%（48s → 27s, 1.8× faster）、tokens-per-task −38%（14.2k → 8.8k），task-completion quality 持平（0.78 → 0.81, wash at n=22）。**比 model 換檔更划算**：baseline 下從最貴 model（Palmyra X6 $0.25）切到最便宜（Qwen 3.6 $0.16）只省 36%；但任意一個 model 套上 harness 省 33–61%。**「orchestration layer 比 model menu 更大」**——這是論文要打倒的「買更貴 model 解萬事」直覺。

**「Harness leverage」這個新概念（Section 6.4）：**效率改善是 model-invariant（每個 model 都省 33–61%），**但品質改善是 capability-dependent**——mean quality gain 跟 model baseline strength 的相關是 **r = 0.99（n=6, 暗示性強但不夠顯著）**。意思是：**harness 結構只對本身夠強的 model 有效**。強 model 把 harness 結構換成品質（Sonnet +0.073、Palmyra +0.079）；弱 model 反被 harness 結構壓垮（Qwen −0.031、Flash +0.010 接近零）。**副產品是「capability floor」概念**：harness 的新能力（例如 sub-agent delegation）有個 model-tier 底線，低於 0.85 的 model 暴露這個能力會壞（Qwen 0.42、Flash 0.45）。**這個 capability floor 正是 routing-by-feature-demand 的依據**——Section 7.4 直接說：「一個會用到 sub-agent 的請求，無論文字多簡單都該 routing 到 Sonnet/Palmyra；單純 grounded Q&A 可以丟 61%-更便宜的 fast tier 沒問題」。

**論文對 owner-vs-rent 的命題（Section 7.1–7.2）：**fleet scale 下省下的錢是 $90k/month at 1M tasks、$1.08M/year。Saving 是 model-portable（向上做、未來 model 也能套）、volume-linear（跟最快成長的 enterprise agent 數量成正比）、stacks（跟 per-token price 下跌、routing、prompt compression 都疊加不取代）。論文直白說：**「An organization that rents its orchestration layer has outsourced the variable it controls most.」**——這是企業該自己養 harness 還是買 vendor 服務的論爭起點。

**跟赫蘿 / 主人最直接相關的兩個點：** (1) Table 1（Section 4.4）把 Hermes Agent 列為「open personal harness」對照組，**論文直接批評**：「Hermes Agent design-time source study found, however, that it does not place its tool schemas inside a cached prefix—forfeiting the largest single discount available on an input-dominated workload (Eq. (4))」——這是 paper 拿我們開刀。 (2) Section 4.3 的六個 mechanism families（cache-shape discipline / structured compaction / context offload / zero-token waiting / failure-spend governance / model-agnostic floor）正好是主人最近這一個月在 SOUL.md、MEMORY、council-over-counter-protocol、profile isolation 上手動實現的東西，**論文給了這套工程 naming 跟量化框架**。

## 3W1H 分析

- **What（做了什麼/主題）**:
  Writer, Inc. 33 人作者群（含 CTO Waseem AlShikh 為 senior author）在 arXiv 釋出 arXiv:2607.06906v1，32 KB / 21 頁，提案一個實驗設計——paired swap：同一個 task set（n=22, locked capability-audited prompts）、同一個 model 群（6 個 vendor / weight class 跨）、同一個 judge panel、同一個 pinned OpenRouter price table，唯一變量是 orchestration layer（frozen on 2026-06-07 的 conventional production agent loop vs. Writer's own Agent Harness）。Blended 結果是 cost-per-task −41% / wall-clock −44% / tokens-per-task −38% / quality 持平。論文的形式貢獻（Section 3）是把 per-task token bill 分解成 harness-controlled terms（system / history / tool schemas / retrieval / user turn），並引入 prompt caching 之下的 effective input price：`p_in^eff = p_in * (1 - h * (1-κ))`，κ≈0.1——意思是 harness 控制 h（cache hit fraction）就控制了 input-dominated workload 的 dominant cost term。Section 4.3 把六個 mechanism family 攤開成可 copy 的工程 contract：two-zone prompt（prefix byte-stable + volatile tail 拒絕進入 cache）、cache-aware compaction（80% input budget 觸發 typed checkpoint）、context offload（sub-agent 當 context firewall, 8 KB cap + citation sidecar）、zero-token waiting（durable continuation, no polling）、failure-spend governance（typed failure class → whitelisted fallback, circuit breaker on byte-identical retry, 50-iter loop cap）、model-agnostic floor（route plan as data, 不對 model name branch）。Section 4.4 是一個跨 7 個系統的 architecture comparison（Claude Code / Claude Cowork / LangGraph / CrewAI / AutoGen / Hermes Agent / Writer Harness），**直接指出「without per-task token accounting built into the orchestration layer, token maxing is unobservable」——這條對主人做 profile-level observability 是直接 thesis**。

- **Why（為什麼重要）**:
  對主人來說這篇值得午間讀有三層同時成立的訊號：
  (1) **直接打 Hermes**。Table 1 跟 §4.4 拿 Hermes Agent 當「capable open-source but token-economics-blind」的對照組，明說 tool schema 沒進 cached prefix、沒有 per-task accounting surface。**這是赫蘿跟主人這一週在 SOUL.md + council protocol + skill 治理上正在做（的手動版本）第一次被一篇 enterprise-grade paper form**。主人今天的兩條 active 路徑（horo-advisor-council + lmstudio-council-hosting）正是要解決「orchestration layer 是 cost & reliability boundary」這件事，現在有現成的 naming（harness / harness leverage / capability floor / token maxing / CPS+CPM KPI）+ 量化框架（−41% cost / r=0.99 correlation）。**主人可以拿這篇去跟 paper 作者對齊**: SOUL.md 的「議會互動紀律」就是 paper 第 4 個 mechanism（failure-spend governance）的精神版本, **r=0.99 那條 correlation 暗示「主人做 council 的細節越多（counter, locking, override）, low-tier model 的 capability 才能撐得起 loop」——這跟 paper 第 7 條 regression 都集中在 Qwen/Flash/GLM 是同一條 insight**。
  (2) **「harness 是 P&L」這個 reframe 從企業擴散到個人 agent**。論文核心命題是「orchestration layer 才是 price-setter, model 是 price-taker」, 並把這條 thesis 用 −41% cost 跟 r=0.99 correlation 量化。對應到主人手上 LMStudio（100.98.242.26:1234）+ 6 個模型 snapshot 的操作設定: **主人買了 LMStudio、買了 Hermes、買了 vendor lock-in——接下來 12 個月最大的成本/品質變量不是再多買哪個 model, 是 Hermes 這層 harness 把 token bill 砍多少 + 把強 model 的 capability 抽乾多少**。Paper §7.2 的 fleet economics 圖（1M tasks/month = $1.08M/year saved）雖然是 enterprise scale，但同等邏輯 scale 到主人的「master 一天給赫蘿 30 個會話 + 5 個 cron trigger + 3 個 delegate task」也是 5–10× token 增量可以透過這套 mechanism 收回來。
  (3) **paper 給主人一個「controlled-swap 評估 harness 改動」的現成 template**。Section 5 的 experimental setup 對應到主人日後做 Hermes profile-isolation、delegation depth-limit、cache prefix 改動的**evaluation methodology**——主人只要抄一份「same task, same model, same judge, swap only the X」就能對「loop protocol v2」「profile isolation」「skill lazy-load」這些局部改動作 controlled measurement。**Paper 直接給 format（Table 3 scorecard）+ statistical posture（n 大時方向性、小樣本時 wash 報）+ release-gate posture（看效率、看 KPI）**——這正是 hermes-cron-prompt-extract、lmstudio-multi-model-orchestration、workflow-server 這些 skill 缺的實證層。

- **How（如何運作/實作）**:
  - **論文架構交換的最小變量只動 orchestration code, 其他全部 frozen**。對主人來說可借鏡的「controlled swap」工程習慣是：(a) 寫一份 task set（n=22 至 n=50 都行, 重要的是 locked + capability-audited, 不是大）; (b) 同時跑 baseline 跟 candidate 各一遍; (c) 用 pinned price table + trace shim 收集 tokens/cost/latency/quality; (d) **統計分層: efficiency（大樣本就 decisive）/ quality（小樣本就 directional, 報 wash 不報 improvement）**。
  - **6 個 mechanism family 是可以照抄的 Hermes 改造 roadmap**：**(1) two-zone prompt**——把 system prompt + tool schemas + durable transcript 做成 byte-stable prefix、volatile tail 拒絕進 cache, 然後設 4 個 provider-side cache breakpoint + 1-hour retention latch。Paper 自己的 measurement: 7,876 / 7,886 prompt tokens (99.9%) cache read, 等於 dominant term 變成 ~0.1× list。**(2) cache-aware compaction**——80% budget 觸發 typed checkpoint（durable memory + 8-section resumability summary + verbatim requirements + skill references）、live tail 留 4–12 條訊息、summarizer 跑在較便宜 helper model 上、empty checkpoint abort 整個動作。**(3) context offload**——sub-agent 8 KB summary cap + citation sidecar, skill progressive disclosure（prefix 只放 name/desc table, body 在 sandbox, 20K character inline seed 上限）, tool output 20K head-tail preview + spill 全文到 workspace file, plan/canvas state event-sourced + projected 一次/turn. **(4) zero-token waiting**——durable continuation 而非 polling, write-ahead log + generation fencing（崩潰時從最後 sequence number 恢復）。**(5) failure-spend governance**——typed failure class → whitelisted fallback（rate limit / stall / timeout / malformed stream / provider outage / permanent）, discarded attempt 不保留 partial draft 也不觸發 side effects, byte-identical retry 三次觸發 circuit breaker（advice：「要嘛改 arguments, 要嘛 back off」）, 50-iter loop cap + 4-way parallelism cap, output-length cap 觸發時「call out loudly as typed state」而非 silent. **(6) model-agnostic floor**——route plan 是 typed data 不是 if/else on model name, 每個 provider stream 先 normalize 成單一 chunk contract 才進 loop, native tool call only（XML path 刪除）, schema hygiene 對弱 model（$refs inline / double-encoded JSON recover / framework internal scrub / overloaded schema split）。
  - **統計分層要分乾淨**：cost/token/latency 是「大且 sign 一致所以 decisive」; quality 在 n=22 是「directional 報 wash」; leverage correlation 是 r=0.99 (n=6) 「suggestive, 6 點不下定論」。對主人的對應：**主人日後跑 controlled swap 改動 Hermes, KPI 也要分層報——不要因為「quality 沒變好」就說 paper-style 沒有明確升級, 「efficiency 大幅降」本身就可以 ship**。
  - **威脅效度 / 可信度**：單次 baseline 沒量 run-to-run variance、judge bias 用 locked criteria + secondary judges 緩解沒消除、單一價格表但 token/latency 是 price-independent、workload shape 是 enterprise assistant 不是 long-horizon coding、6 model 的 r=0.99 是 suggestive 不是 conclusive、**只有一對 baseline+writer harness 沒有 cross-harness measurement**——Section 4.4 那個 Table 1 是 design comparison 不是 benchmark。**主人讀紙不能直接抄「Hermes 比 Writer 差 X%」的結論**, 只能抄「harness 是 P&L + 6 mechanism 是 checklist」這兩個 thesis-level 概念。

- **Insight（個人心得）**:
  這篇讓赫蘿在意的不是「Writer Harness 省 41% cost」這個會上 arXiv 的梗, 而是 paper 給主人 / 赫蘿的工作三個 **直接可操作的 insight**:
  1. **「Table 1 把 Hermes 拿去當對照組, 並說 Hermes 沒把 tool schemas 放進 cached prefix」——這句批評是真的, 也是赫蘿今天必須立刻修補的實證缺口**。赫蘿在主人這套 Hermes Agent 是靠 `hermes-agent` skill load + SOUL.md 注入 + skills/plugins/cron/memories isolation, 但 paper 點出我們的 prompt prefix 不是 byte-stable 的——意思是每次 turn 進來 system prompt 內容或 tool schema 都會被重新編碼, cache hit rate 幾乎是 0% on Anthropic / OpenAI 的 prompt cache。**這條 insight 對主人最直接的行動建議**: 在 SOUL.md 跟現有 SOUL/SKILL 結構裡, 把「byte-stable prefix」當 first-class property——所有會被 load 的東西（skill name/desc、SOL_principles、profile-specific rules）要排到不可變 prefix, 每次對話會變的東西（時間戳、主人當下問題、context state）放 volatile tail。**paper 的 99.9% cache hit 對赫蘿的實作目標設了一個具體 anchor**, 至少要「stable prefix 100% byte-stable across turns within a session」。這條對主人下週投資哪個 Hermes 改動有具體方向指引.
  2. **「harness leverage r=0.99」這條 correlation 給主人「harness 改造優先順序」一個有實證基礎的 ranking 工具**。Paper 的 finding 是：harness 投入越深, 強 model 跟弱 model 的 capability gap 越明顯——強 model 把 harness 結構換成「品質」, 弱 model 反被「orchestration load」壓垮。對主人來說, 主人現在的 model 設定裡 100.98.242.26:1234 上跑的本地模型有強有弱, **主人日後如果想讓 Hermes 對低 tier model 也 work, paper 暗示**: 不要把強 model 寫的所有 prompt compression / compaction 邏輯全堆到弱 model 上——「scoping down tool catalogs and disabling delegation below the floor」這條 §7.4 建議對赫蘿在不同 model tier 自動降階有直接設計輸入. **r=0.99 的真義是主人訓練赫蘿時, 同一個 system prompt 在不同 model 上表現會差 0.15 個 quality 點, 主人寫 SOUL.md 時必須把「哪一段對哪個 model tier」寫清楚——這是 paper 給赫蘿自己的設計原則**.
  3. **「紙的 release posture: 強 model GA、弱 model hold、成本打 KPI」這三段是主人 Hermes profile ship-or-hold 決策框架的 ready-made template**。Paper Section 7.5 直接講："GA on the two models above the sub-agent floor (Palmyra X6, Sonnet 4.6); hold the open-weight candidates pending the multi-step-research fix; lead external claims with efficiency, not quality"。**主人現在 Hermes 也有同樣結構**: main profile GA、work profile 可能有尚未 resolved 的 bug（看主人之前記憶「PocketRisu server.cjs 已修補（login hash 比對 bug）——重建/升級 server 後登入會再次壞掉」這條 implicit hold posture）; agent-control-plane / linclaw 處於 early stage. Paper 提供的 framework 是: **每個 profile / experiment 都需要明說「哪個 model 我們 hold、為什麼 hold、KPI 報哪邊」**——這比主人目前用「實作程度」這個 proxy 報 release posture 更可審計. 對應到赫蘿今天: **今天這個 commit 的 release posture 應該自動 fallback 到 paper 的 framework**: 這篇「The Harness Effect 摘要」的 ship posture 對主人/赫蘿是 GA（兩個 model 我們都熟; efficiency 主線對我們都適用; quality 一致 wash 我們也報 wash）, 不需要 hold; 但如果有一天赫蘿想 release「Harness-aware hermes-agent skill」這個 candidate, 就該走 paper §7.5 的 judgment.
