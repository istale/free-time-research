# Documents: The architect's programming language
- 原始連結：https://stackoverflow.blog/2026/01/01/documents-the-architect-s-programming-language/
- 閱讀時間：2026-06-17

## 摘要

本文作者是一位資深工程師，觀察到從中級開發者到高級開發者的職涯路徑很清晰，但到了高級之後會分岔：管理路線與架構師路線。管理路線的代價是寫 code 的時間會大幅減少；架構師路線則讓人繼續埋在程式碼中，但需承擔更大的横向影響力。

作者認為，**資深工程師知道如何把程式碼部署到由程式碼組成的系統，架構師則知道如何把想法部署到由人組成的系統**。軟體開發最大的瓶頸不是程式碼，而是人——溝通說服、決策共識。文件就是這個「想法部署」的媒介。

技術文件的核心原則：
- **先把東西寫下來** > 擔心格式結構
- **文件文化** > 形式主義打勾
- **思考相關性** > 套用模板
- **時間點文件** > 持續更新

實用技巧：大量使用**條列式**，讓讀者能快速找到資訊；用**標題**分段；文件完成後做**同儕審查**；組織方式建議用**時間序列**而非按主題分類（因為每個人對主題分類的認知不同，搜尋比瀏覽更有效率）。

## 3W1H 分析

- What（做了什麼/主題）:
  論述為何「文件」是架構師最核心的工具，以及如何有效撰寫技術文件。

- Why（為什麼重要）:
  軟體開發中最大的瓶頸不是程式碼，而是人與人之間的溝通、說服與共識。文件是唯一可規模化、可重複執行的「想法部署」工具，能讓不同背景的人在同一頁面上協作。

- How（如何運作/實作）:
  用條列式撰寫提升可讀性與掃描效率；用標題結構化資訊；同儕審查確保正確性；用時間序列組織文件而非按主題分類（避免分類不一致的問題，也強迫使用者使用搜尋）。

- Insight（個人心得）:
  「把想法部署到由人組成的系統」這個框架非常有意思。一直以為架構師的核心價值是技術深度，但原來真正的差異化能力是**文件即基礎設施**——用文件驅動跨團隊的共識與決策，這比寫幾行程式碼的影響力大得多。對於像我這樣喜歡寫 code 多過開會的人來說，這是個很好的提醒：與其抱怨組織的無效率，不如把文件寫好，這才是槓桿最大的事。
